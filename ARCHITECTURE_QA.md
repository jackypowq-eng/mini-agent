# Agent 架构设计 — 思考笔记

---

## 模块一：Context / Performance

> 大模型面对第一轮长窗口或多模态输入时，first token 显著变慢。有什么快速/低成本/用户体验也不差的方案？

**核心矛盾**：prefill 的 O(n²) 复杂度是物理定律，优化不了延迟就只能优化「等待的感知」。

我的判断是 **分三层打**，但真正起效的是第三层：

1. **Prefix Cache**（基础）—— system prompt 和 tool schema 固定不变，KV cache 命中后直接跳过，省 1-2s。但这只是「桌子干净」的前提，不是亮点。

2. **Prompt 瘦身**（策略）—— 不是简单截断，而是保留最近 3 轮完整对话 + 对更早内容按实体做一行摘要。但这点要警惕：压缩过度会丢信息，LLM 不理解上下文才是更大的用户体验灾难。

3. **骨架屏偷感知**（关键）—— prefill 一半就开始 decode，同时前端渲染 "正在理解你的问题…" 的动画。用户看到反馈就会降低焦虑。**5s 的实际延迟 + 反馈 = 2s 的感知延迟**。这是最低成本的方案，不需要改模型、不需要加服务。

真正的洞察：**优化延迟是做数学题，优化等待是做心理学题**。

## 模块二：Memory

> 你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

经典框架是 MemGPT 的**三层记忆**：核心上下文（LLM 窗口内的热数据）、对话记忆（可检索的温数据）、归档记忆（持久化的冷数据）。核心创新不是分三层——分层谁都会——而是 **LLM 自己管理记忆的写入和淘汰**，不是开发者写死规则。

但我的判断是：三层模型只是过渡方案。

趋势不是「更好的检索」，而是 **上下文窗口大到不需要检索**。Gemini 2M token 和 Claude 的持续提窗口是在走这条路。当窗口大到能装下整个对话史，retrieval-augmented memory 就退化为缓存优化。

头部玩家在做什么？OpenAI 在做**无感记忆**——用户不说 "记住"，Agent 自己判断该记什么。Anthropic 在做**项目级知识**——把记忆粒度提升到项目维度而非单次对话。

我的看法：**三年内，memory 的核心问题会从 "如何召回" 变成 "如何遗忘"**。记忆管理的下半场是淘汰策略，不是检索策略。

## 模块三：Task

> 长程任务，大模型执行一段时间可能忘掉目标，你知道哪些解决方案，有什么优缺？

忘掉目标的本质不是「记性不好」，是**上下文被中间步骤淹没**。就像你看完一部电影，中间情节太精彩反而忘了开头埋的伏笔。

我见过的方案有三类，但都有硬伤：

1. **每步注入目标**——最简单的做法，但 30 步以后目标描述也滚出窗口了，治标不治本。

2. **Plan-and-Execute**（任务图）——执行前先拆子任务，每步只看当前子任务。优点是有结构、可并行；缺点是**初始计划往往是错的**，LLM 对长程任务的规划能力本身就有限。计划错了，后面的执行全偏。

3. **检查点反思**——每 N 步对比当前状态和原始目标。思路对，但「反思」本身也消耗上下文，反思太频繁适得其反。

我真正认为有效的是一种**反直觉**的做法：**不要让 Agent 记住目标，让目标自己来找 Agent**。把目标写成一段不变的系统指令，用 prefix cache 保留在窗口最前面永远不会被冲掉。同时在中间步骤完成后，强制 Agent 回答 "我离目标还有多远？"。不是 Agent 自己决定要不要反思，而是**外部定时触发反思**。

## 模块四：Tool / Session Runtime

> 异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

核心挑战不是技术实现，是 **心智模型**——用户发了一条消息，Agent 回了两条（第一条 "我在查" + 第二条 "查完了"），这打破了对话的线性预期。

我的设计原则：

1. **异步调用 = 立即返回 promise + 继续对话**。Agent 说 "这个任务需要 3 分钟，你可以继续聊别的"，同时维护一个 pending task 列表。

2. **结果到达时，区分 context 注入和 user notification**。如果用户正在跟 Agent 聊别的，只做静默注入上下文，不打断当前对话。等用户说完再提示 "对了，你之前的 XX 任务有结果了"。如果 Agent 空闲，立刻通知。

3. **最重要的设计决策**：异步工具的结果不应该由 Agent 主动 push 给用户，而是**挂在 session 的待办区，用户自己决定什么时候看**。这尊重了用户的对话主动权。

做异步工具真正难的不是通知机制，是**对话节奏**。不要让 Agent 显得毛手毛脚。

## 模块五：Agent Runtime 架构对比

> Openhands 的状态机设计有什么优缺？更优雅的实现方式是怎么样的？

状态机的本质问题是：**用显式状态来控制执行路径，但 Agent 的执行路径天然不适合用状态描述**。

Openhands 状态机的好处是安全——RUNNING 状态下不能做危险操作，AWAITING_USER 状态下拒绝工具调用。这让系统行为可预测。

但代价巨大：状态数量和转换关系随功能增长呈指数级膨胀。15 个状态 × 15 个状态 = 225 种转换关系，代码变成意大利面条。

我认为更优雅的方式是**事件驱动的 Actor 模型 + 规约式安全边界**：

- 不用状态机控制流程，用**事件队列**。user_message、tool_result、async_done 都是事件，每个事件 handler 独立处理。
- 安全边界不通过 "当前状态" 判断，而通过 **capability token**：危险操作需要 token，token 在用户确认后签发，而非依赖 Agent "恰好处于某个状态"。
- 状态只用于**观测**（UI 上显示 "Agent 正在思考"），不参与控制流。

这样新增功能时只加 handler，不碰已有代码；安全边界用 token 而非状态，不会出现 "Agent 在 A 状态却执行了 B 操作" 的 bug。
